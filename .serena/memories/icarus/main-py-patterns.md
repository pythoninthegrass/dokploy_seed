# Main.py Architectural Patterns for Icarus

## 1. APP CREATION (setup flow)

### Entry Point: `cmd_setup()` (line 833)
- **State Check**: Fails if state file already exists (prevents duplicate setup)
- **Project Creation First** (lines 843-851):
  - API call: `project.create` with name + description
  - Stores: `projectId`, `environmentId` in state
  - Early state save (line 904) - allows destroy to cleanup if later steps fail

### State Structure
```json
{
  "projectId": "proj-xyz",
  "environmentId": "env-123",
  "apps": {
    "app_name": {
      "applicationId": "app-456",      // non-compose
      "appName": "app-appname",        // computed by Dokploy
      // OR
      "composeId": "comp-789",         // compose
      "source": "compose"              // optional marker
    }
  }
}
```

### App Creation Pattern (lines 870-901)
Two paths based on source type:

**Non-Compose Apps** (lines 893-901):
```python
result = client.post("application.create", {"name": name, "environmentId": environment_id})
state["apps"][name] = {
    "applicationId": app_id,
    "appName": app_name
}
```

**Compose Apps** (lines 873-890):
```python
result = client.post("compose.create", {
    "name": name,
    "environmentId": environment_id,
    "composeType": app_def.get("composeType", "docker-compose")
})
state["apps"][name] = {
    "composeId": compose_id,
    "appName": app_name,
    "source": "compose"
}
```

### Resource Configuration Steps (lines 906-1032)
In order:
1. **Providers** (lines 906-943): Docker/GitHub configuration
2. **Command Overrides** (lines 945-959): Uses `resolve_refs()` for placeholders like `{other_app}`
3. **Domains** (lines 961-975): Calls `domain.create` via `build_domain_payload()`
4. **App Settings** (lines 977-986): autoDeploy, replicas via `application.update`
5. **Volume Mounts** (lines 988-1000): `mounts.create` via `build_mount_payload()`
6. **Ports** (lines 1002-1016): `port.create` via `build_port_payload()`, stores portId
7. **Schedules** (lines 1018-1032): `schedule.create` via `build_schedule_payload()`, stores scheduleId

### State Storage Pattern
Each resource type stores a lookup dict indexed by a key:
- **Ports**: `state["apps"][name]["ports"][publishedPort] = {"portId": ...}`
- **Schedules**: `state["apps"][name]["schedules"][name] = {"scheduleId": ...}`
- **Mounts**: `state["apps"][name]["mounts"][target] = {"mountId": ...}` (on redeploy)
- **Domains**: `state["apps"][name]["domains"][host] = {"domainId": ...}` (on redeploy)

## 2. APP DESTRUCTION (destroy flow)

### Entry Point: `cmd_destroy()` (line 2014)
1. Load state
2. Cleanup stale Traefik routes via SSH
3. Delete project (cascades to all apps): `project.remove` with projectId
4. Delete state file: `state_file.unlink(missing_ok=True)`

**Key**: Only needs projectId - deletion is cascading. No per-app deletion needed.

## 3. APP STATUS (status command)

### Entry Point: `cmd_status()` (line 1192)
Iterate over `state["apps"]` and fetch remote status:

```python
for name, info in state["apps"].items():
    if info.get("source") == "compose":
        comp = client.get("compose.one", {"composeId": info["composeId"]})
        status = comp.get("composeStatus", "unknown")
    else:
        app = client.get("application.one", {"applicationId": info["applicationId"]})
        status = app.get("applicationStatus", "unknown")
    print(f"  {name:10s}  {status}")
```

**Pattern**: Uses `application.one` (non-compose) or `compose.one` (compose) with the stored ID.

## 4. STATE FILES & ID STORAGE

### State File Location
- Path: `.dokploy-state/{env_name}.json`
- Generated by: `get_state_file(repo_root: Path, env_name: str) -> Path`
- Directory created on demand: `save_state()` calls `parent.mkdir(parents=True, exist_ok=True)`

### Load/Save Pattern
- **Load**: `load_state(state_file: Path) -> dict`
  - Exits if missing (assumes setup must run first)
- **Save**: `save_state(state: dict, state_file: Path, *, quiet: bool = False) -> None`
  - Pretty-prints JSON with indent=2
  - Optional `quiet` flag to suppress "State saved" message

### State Validation
- `validate_state(client: DokployClient, state: dict) -> bool`
- Checks if project still exists on server: `client.get("project.all")`
- Returns True if valid OR if server unreachable (assume valid)
- Returns False only if project confirmed gone

### Early State Saves
- After project+app creation (line 904), before provider config
- Allows `destroy` to clean up if setup fails mid-way

## 5. YAML CONFIG PARSING

### Entry Point: `load_config(repo_root: Path) -> dict`
- Path: `repo_root / "dokploy.yml"`
- Parser: `yaml.safe_load()`
- Validation happens after load (separate step)

### Validation Steps
1. **`validate_config(cfg: dict)`** (line 100):
   - Checks env_targets reference valid apps
   - Checks deploy_order references valid apps
   - Checks GitHub config exists if any app has source="github"

2. **`validate_env_references(cfg: dict)`** (line 121):
   - Ensures environment overrides reference apps that exist

### Environment Merging
- **`merge_env_overrides(cfg: dict, env_name: str) -> dict`** (line 133):
  - Deep copy config
  - Pop out `environments` block
  - Merge `environments[env_name].github` overrides into `github`
  - Merge `environments[env_name].apps` overrides into each app definition
  - Returns merged config

**Pattern**: Overrides done with `.update()` on copied objects - original untouched.

## 6. API CALL PATTERN

### DokployClient Class (line 658)
```python
class DokployClient:
    def __init__(self, base_url: str, api_key: str):
        self.client = httpx.Client(
            base_url=base_url.rstrip("/"),
            headers={"x-api-key": api_key},
            timeout=60.0,
        )
    
    def get(self, path: str, params: dict | None = None) -> dict | list:
        resp = self.client.get(f"/api/{path}", params=params)
        resp.raise_for_status()
        return resp.json()
    
    def post(self, path: str, payload: dict | None = None) -> dict:
        resp = self.client.post(f"/api/{path}", json=payload or {})
        resp.raise_for_status()
        if not resp.content:
            return {}
        return resp.json()
```

### API Call Conventions
- **GET**: `client.get("endpoint.method", params={"key": value})`
- **POST**: `client.post("endpoint.method", {"key": value})`
- Path format: `"endpoint.method"` (e.g., `"application.create"`, `"domain.byApplicationId"`)
- Response unwrapping: `resp.json()` automatically (handles empty responses)
- Errors: `.raise_for_status()` raises `httpx.HTTPStatusError` on bad status

### Payload Builders
Helper functions construct payloads before API calls:
- `build_domain_payload(resource_id, dom, *, compose=False)` - line 298
- `build_mount_payload(app_id, mount)` - line 336
- `build_port_payload(app_id, port)` - line 351
- `build_schedule_payload(app_id, sched)` - line 362
- `build_app_settings_payload(app_id, app_def)` - line 324
- `build_github_provider_payload(app_id, app_def, github_cfg, github_id)` - line 230
- `build_build_type_payload(app_id, app_def)` - line 261

**Pattern**: Builders validate input and return clean payloads. Some handle optional fields with conditional assignment.

## 7. RECONCILIATION PATTERN (Redeploy)

Used in `cmd_apply()` when state exists and is valid (line 1181-1186):
```python
if is_redeploy:
    cleanup_stale_routes(state, cfg)
    reconcile_app_domains(client, cfg, state, state_file)
    reconcile_app_schedules(client, cfg, state, state_file)
    reconcile_app_mounts(client, cfg, state, state_file)
    reconcile_app_ports(client, cfg, state, state_file)
```

### Reconciliation Functions
All follow the pattern: **fetch remote → compare with desired → create/update/delete**

Example: `reconcile_ports()` (line 492):
1. Fetch existing: `existing_by_port = {p["publishedPort"]: p for p in existing}`
2. Build desired: `desired_by_port = {p["publishedPort"]: p for p in desired}`
3. For each desired:
   - If exists: check for diffs, POST update if changed
   - If new: POST create, store response ID in state
4. For each existing but not desired: POST delete
5. Return state dict mapping key → {"id": ...}

### Wrapper Functions
- `reconcile_app_domains()` (line 628): Iterates apps, calls `reconcile_domains()` per app
- `reconcile_app_schedules()` (line 422): Iterates apps, calls `reconcile_schedules()` per app
- `reconcile_app_mounts()` (line 560): Iterates apps, calls `reconcile_mounts()` per app
- `reconcile_app_ports()` (line 534): Iterates apps, calls `reconcile_ports()` per app

Each wrapper:
- Loads state fresh (may reload multiple times in cmd_apply)
- Updates state dict with new results
- Only saves state if anything changed
- Handles both compose and non-compose apps

## 8. REFERENCE RESOLUTION

### Function: `resolve_refs(template: str, state: dict) -> str` (line 154)
Replaces `{app_name}` placeholders with `state["apps"][app_name]["appName"]`:
```python
def resolve_refs(template: str, state: dict) -> str:
    def replacer(match: re.Match) -> str:
        ref = match.group(1)
        if ref in state["apps"]:
            return state["apps"][ref]["appName"]
        return match.group(0)  # leave unresolved as-is
    return re.sub(r"\{(\w+)\}", replacer, template)
```

**Usage**:
- In `cmd_setup()`: Command overrides (line 953)
- In `cmd_env()`: Filtered env vars and custom env (lines 1067, 1103)
- In `cmd_plan()`: Same as env (lines 1407, 1411)

**Pattern**: Leaves unresolved refs unchanged (doesn't fail).

## 10. PLAN COMMAND

### Entry Point: `cmd_plan()` (line 1752)
- Calls `compute_plan()` to generate list of changes
- Calls `print_plan()` to display terraform-style output

### `compute_plan()` Logic (line 1679)
1. If no state: `_plan_initial_setup()`
2. If state exists but invalid: `_plan_initial_setup()`
3. If state valid: `_plan_redeploy()`

### Change Structure
```python
{
    "action": "create|update|destroy",
    "resource_type": "project|application|compose|domain|environment|settings|mount|port|schedule",
    "name": str,
    "parent": str | None,  # app name if child resource
    "attrs": dict,
}
```

### `_plan_initial_setup()` (line 1221)
Creates change entries for:
- Project
- Apps (application or compose)
- Domains (per app)
- Settings (autoDeploy, replicas)
- Mounts
- Ports
- Schedules
- Environment vars (filtered + custom)

### `_plan_redeploy()` (line 1371)
Compares remote state to desired config for:
- Environment vars (key diff: added/removed)
- Domains (per app)
- Mounts (per app)
- Ports (per app)
- Schedules (per app)

**Pattern**: Diffs compute (old, new) tuples or lists of added/removed keys.

## KEY TAKEAWAYS FOR DATABASE SUPPORT

1. **State Structure**: Store resource IDs per database (index by name/key, store ID)
2. **Create Pattern**: Build payload → POST → extract ID → store in state
3. **Reconcile Pattern**: Fetch existing → index by key → diff desired → create/update/delete
4. **Redeploy**: Call wrapper function that iterates apps, calls base reconcile, saves state
5. **Destroy**: Cascade delete via parent (don't need per-db delete)
6. **Status**: Fetch remote via ID stored in state
7. **Config Parsing**: Load YAML → validate references → merge overrides → use
8. **API Pattern**: Thin httpx wrapper with .get(path, params) and .post(path, payload)
9. **Payload Builders**: Helper functions that handle optional fields and conditional logic
10. **Early Saves**: Save state after risky operations to enable partial cleanup on failure
